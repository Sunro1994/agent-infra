## 비고

- T-001 happy path 는 HTML5 `required` 가 빈 이메일 submit 을 막아 JS 핸들러까지 도달하지 못한다. 사양상의 "빨간색 메시지" 를 검증하기 위해 `required` 및 `type=email` 속성을 콘솔에서 제거한 뒤 동작을 확인했다. 운영 환경에서 JS 가 비활성화되거나 `required` 가 우회되면 검증이 트리거된다.
- T-001 KD-1 재현 단계도 `type=email` 을 `text` 로 바꾼 뒤 진행했다 (HTML5 가 `a@b` 를 먼저 차단하기 때문). 이는 곧 다중 방어선이 부재함을 보여준다 — HTML5 한 겹이 뚫리면 JS 정규식은 사실상 무방비.
- T-003 KD-3 직후 콘솔에 `404 /x` 가 남는다 (XSS payload 의 `<img src=x>` 로드 시도). 이는 XSS 실행 흔적이다.
- 시나리오 외 우려: 서버 측 검증 부재. 현재 페이지는 정적 HTML 이며 어떤 형태의 백엔드 검증도 보이지 않는다. 클라이언트 검증을 강화하더라도 서버 검증이 추가되지 않으면 실효성이 낮다.
